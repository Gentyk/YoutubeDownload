## Problem Statement

Личный инструмент для скачивания аудио из YouTube видео в формате mp3. Принимает один URL или несколько URL через запятую. Файл сохраняется с именем = название видео. После скачивания доступна красочная страница со статистикой: сколько скачано, скорости скачивания (постфактум), история по дням.

Это **personal use** — не для других людей, не для продажи, не для масштабирования. Цель — удобный персональный workflow.

## What Makes This Cool

- **Одна команда, паста URL, готово** — `python app.py`, открывается браузер, вставляешь список URL, жмёшь Download, идёшь пить кофе.
- **Дашборд "красочно"** — не просто лог, а Chart.js графики: скачано по дням, средняя скорость, top-каналы, histogram по длительности.
- **Все имена файлов правильные** — никаких `Rick_Astley_Never_Gonna_Give_You_Up_HD.mp3`, yt-dlp подставляет реальное название (`Rick Astley - Never Gonna Give You Up.mp3`).
- **Persistent история** — SQLite база, всё видно за всё время.

## Constraints

- **Личное использование** — только локально, без auth, без деплоя
- **Python** (yt-dlp нативный, ffmpeg вызывается через yt-dlp)
- **Должен быть Docker-free** для запуска (это персональный тул, не сервис)
- **ffmpeg** должен быть установлен в системе (системное требование)
- **Одна команда запуска**: `python app.py` или `uv run app.py`

## Premises (agreed)

1. **Язык — Python**, не Go. yt-dlp написан на Python, есть прямой API без shell-out. Go дал бы лишний subprocess overhead и боль с парсингом JSON прогресса.
2. **Конвертация в mp3 — через yt-dlp postprocessor**, который сам дёргает ffmpeg. Нужен ffmpeg в PATH.
3. **Статистика — post-hoc**, не live. После каждого скачивания записываем строку в SQLite, дашборд читает базу и рендерит графики.
4. **Хранилище — SQLite**, не Postgres/JSONL. Один файл `yt2mp3.db`, легко бэкапить.
5. **UI — FastAPI + HTMX + Tailwind**. Свой контроль над дизайном, "вау" дашборд возможен.

### Модель данных (SQLite)

```sql
CREATE TABLE downloads (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    url TEXT NOT NULL,
    video_id TEXT,
    title TEXT,
    channel TEXT,
    duration_s INTEGER,
    file_size_bytes INTEGER,
    download_time_s REAL,
    avg_speed_mbps REAL,
    file_path TEXT,
    status TEXT,             -- 'success' | 'failed'
    error TEXT,
    started_at TIMESTAMP,
    finished_at TIMESTAMP
);

CREATE INDEX idx_finished_at ON downloads(finished_at);
```

### API endpoints

| Method | Path | Назначение |
|--------|------|------------|
| GET    | `/`              | главная страница: форма URL + queue |
| POST   | `/download`      | принимает URL через запятую, кикает downloads |
| GET    | `/queue`         | HTMX poll: текущее состояние очереди |
| GET    | `/stats`         | дашборд страница |
| GET    | `/api/stats`     | JSON: данные для Chart.js |
| GET    | `/file/{id}`     | скачать готовый mp3 (если хочется через браузер) |

### Поток скачивания

1. Пользователь пастит URL через запятую в textarea на `/`
2. POST `/download` парсит, разбивает по запятой, очищает
3. На каждый URL — Python `Thread` (или `asyncio.create_task`) запускает yt-dlp
4. yt-dlp callback пишет в in-memory queue + finalizes row в SQLite по завершении
5. Frontend через HTMX каждые 2с poll-ит `/queue` и обновляет таблицу
6. После завершения — кнопка "Show stats" ведёт на `/stats`

### Дашборд (`/stats`) — что показать "красочно"

- **KPI карточки** сверху: всего скачано, общий размер, общее время, средняя скорость
- **Line chart**: скачивания по дням (последние 30 дней)
- **Bar chart**: top-10 каналов по числу скачиваний
- **Histogram**: распределение скоростей скачивания (Mbps)
- **Histogram**: распределение длительности видео
- **Table**: последние 20 скачиваний с пагинацией

Все графики через Chart.js, цветовая палитра — несколько оттенков индиго (типичный современный дашборд).

## Open Questions

1. **Лимит на одновременные скачивания?** Рекомендую 3 — больше упирается в YouTube throttle. Можно вынести в env-var.
2. **Что делать с дубликатами?** Если URL уже скачивали — пропускать или перескачивать? Рекомендую "skip if exists, log it" с галочкой "force re-download".
3. **Playlist URL?** yt-dlp умеет разворачивать плейлисты в список видео. Поддерживать или явно отказывать? Рекомендую поддерживать (одна галочка `--yes-playlist`).
4. **Где хранить downloads?** Сейчас в папке проекта. Сделать настраиваемым через env-var `YT2MP3_DOWNLOAD_DIR`?

## Success Criteria

- Запуск одной командой: `uv run app.py` → браузер открывает `localhost:8000`
- Скачивание одного видео: пастишь URL, жмёшь Download, через 30 сек mp3 в `downloads/`
- Скачивание 5 URL батчем: пастишь через запятую, всё уезжает в очередь, не блокирует UI
- Дашборд на `/stats` показывает реальные данные после первой партии скачиваний
- Имя файла = реальное название видео из YouTube
- Никаких UI-фризов даже если 10 видео в очереди

### Filename sanitize spec (D8 detail)
Strip only: `/`, `\`, `..`, null bytes, control chars (`\x00-\x1f`). **Preserve Unicode/Cyrillic.** Do NOT use yt-dlp `restrictfilenames=True` (it converts Cyrillic → ASCII). Example: `Пятница - Я слышу тебя.mp3` stays as-is.

## Autoplan Phase 3 — Eng Decisions (2026-05-24)

Initial eng score: 5.5/10 → projected 8.5/10. 29 findings, 25 mechanical accepts, 4 taste surfaced.

### Architecture (LOCKED CHANGE)
4 modules instead of 3:
- `app.py` — FastAPI routes only, no business logic
- `queue.py` — **NEW**, JobQueue (ThreadPoolExecutor + RLock + in-memory job dict)
- `downloader.py` — pure function, fresh ydl_opts per call
- `db.py` — conn-per-thread DAL with PRAGMA on every connect

### Locked (auto-decided)
- Bounded queue (`maxsize=50`), 429 on overflow
- Per-job cancellation via `threading.Event` + `POST /cancel/{job_id}`
- Conn-per-thread + `busy_timeout=5000` + `synchronous=NORMAL` + `foreign_keys=ON`
- Single critical section: DB-insert + remove-from-queue atomic
- Fresh `ydl_opts` per call, ctx mgr, O(1) progress hook under lock
- Extended sanitize: trailing dots/spaces strip, 200-byte UTF-8 truncate, empty title → video_id fallback, collision `_2` suffix
- 12-test pyramid (6 unit + 5 integration + 1 smoke): см. test plan artifact
- Security: `Path.resolve` + `is_relative_to` on `/file/{id:int}`, video_id regex `^[A-Za-z0-9_-]{11}$`, SSRF guarded at `re.findall` allowlist step, DNS rebinding Host header check
- Server-side `/api/stats` aggregation: `SELECT date(finished_at), COUNT(*) GROUP BY 1`
- First-run: uv-only check, port-in-use catch, yt-dlp import probe, ship `.gitignore`
- HTMX playlist: two-phase POST with hash-keyed pending state
- yt-dlp postprocessor_hooks separate from progress_hooks (track converting state)
- Lifespan ordering: orphan cleanup → executor start
- Infra: structured logging + yt2mp3.log file, FastAPI lifespan ctx, schema_version migrations, UUID job-id (not auto-inc), pinned HTMX+Tailwind versions, signal handlers (SIGTERM flushes DB + cancels jobs)
