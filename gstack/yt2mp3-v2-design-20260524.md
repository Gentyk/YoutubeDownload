## Problem Statement

v1 работает: личный архив, скачивание YouTube → mp3, дашборд, деплой на 93.115.240.38, login admin. Из реального использования вылезли семь дыр в UX — нет встроенного прослушивания, нет нормальной библиотеки скачанного, нет удаления, всё свалено на одной странице «Скачать», `/stats` смешана с менеджмент-кнопками. v2 закрывает эти семь дыр и оставляет идеологию v1 (single-user, личный архив).

## What Makes This Cool

- **Слушаешь треки прямо на сайте** — паста URL, через 30 сек жмёшь play на той же странице, не идёшь в Finder
- **Библиотека по дням** — листаешь как Apple Photos: «вчера», «23 мая», «декабрь 2025» — видишь свою аудио-историю
- **`/stats` становится чистым отчётом** — без шумных кнопок «скачать», ровный визуальный дашборд, плюс новая аналитика «с каких устройств чаще качаешь»
- **Inline-completion** — на странице «Скачать» внизу появляется блок «только что готово» с плеером, не надо переключаться

## Constraints

- Single-user (login admin), как в v1. Никакого мультиюзера.
- Архитектура остаётся: FastAPI + HTMX + Jinja2 + SQLite + vendored Tailwind/Chart.js
- Деплой не меняется — `docker compose up -d --build` подхватывает всё
- Никаких новых внешних сервисов в v2. Я.Диск откладывается на v2.1

## Premises

Зафиксированы и подтверждены пользователем:

1. **Per-IP — это аналитика моих устройств** (телефон/комп/работа), не разделение на разных людей. Учётка одна.
2. **Библиотека = новая вкладка «Загружено»**. Из `/stats` уезжают кнопки «скачать» — статистика становится чисто отчётной.
3. **Мини-блок «Загружено» внизу `/`** показывает последние 5 успешных, с плеером + кнопкой скачать. Удаление — только из полной вкладки.
4. **Удаление = file unlink + soft-delete в DB.** Файл физически удаляется, строка остаётся с `deleted_at`, чтобы статистика прошлого не сломалась.
5. **Плеер — нативный HTML5 `<audio controls>`.** Никаких canvas-вейвформ. Хром, Safari, Firefox умеют рисовать сами.
6. **Группировка по дням** через SQL `GROUP BY date(finished_at)`. Заголовки: «сегодня», «вчера», «23 мая 2026», «декабрь 2025».
7. **Я.Диск — v2.1**, не v2. Отдельным релизом после стабилизации.

## Approaches Considered

### Approach A: Минимальная v2, Я.Диск откладывается (ВЫБРАН)
~400 LOC, 30-60 мин Claude Code. Все требования 1, 2, 3, 4, 6, 7. Я.Диск в v2.1.
**Pros:** маленький риск, быстрый релиз, можно сразу пощупать UX в браузере.
**Cons:** возвращаемся к Я.Диску отдельно — будет ещё одна сессия и деплой.

### Approach B: Полная v2 через WebDAV
~700 LOC, 1-2 часа Claude Code. То же + Я.Диск через webdav.yandex.ru + app-password.
**Pros:** всё в одном релизе, один деплой.
**Cons:** больше risk-surface, нужно тестировать против реального Я.Диска, увеличивает время до фидбэка от UX-части.

### Approach C: Полная v2 + Yandex OAuth
~900 LOC, 2-3 часа Claude Code. То же + OAuth-flow в Я.Диск.
**Pros:** идеальный UX («Войти через Яндекс»).
**Cons:** требует регистрации приложения в Yandex OAuth, redirect URI (а у нас голый HTTP пока без домена/HTTPS), token refresh, в 2-3 раза больше кода чем WebDAV. Overkill для личного use.

## Recommended Approach

**Approach A.** Личная утилита, цикл «изменил → задеплоил → попробовал» важнее «всё в одном». Я.Диск как отдельный релиз даст шанс посмотреть UX основной части без шумовой загрузки Я.Диск ошибками.

## Architecture changes

### Новые модули
- `yt2mp3/library.py` — отдельные view-helpers для группировки по дням + soft-delete

### Изменения в существующих
- `yt2mp3/db.py`:
  - schema v2 migration: `ALTER TABLE downloads ADD COLUMN client_ip TEXT`
  - schema v2 migration: `ALTER TABLE downloads ADD COLUMN deleted_at TIMESTAMP`
  - новый запрос `get_library_by_day(conn, exclude_deleted=True)` → группирует по `date(finished_at)`
  - новый запрос `ip_breakdown(conn, days=30)` → COUNT GROUP BY client_ip
- `yt2mp3/queue.py`:
  - `_persist_terminal` теперь принимает `client_ip` через job и пишет в строку
- `yt2mp3/app.py`:
  - `JobQueue.submit(...)` принимает `client_ip` (берётся из `request.client.host` в `/download`)
  - **новый route** `GET /library` — страница с группировкой по дням
  - **новый route** `GET /library/fragment` — HTMX обновление для inline-mini-panel на `/`
  - **новый route** `POST /file/{id}/delete` — soft-delete + unlink файла (только для success-rows)
  - `/stats` теряет ссылку «скачать» в last-20 таблице → остаётся только название и метаданные
  - `/api/stats` дополняется ключом `by_ip` (последние 30 дней)

### Шаблоны
- `templates/base.html` — добавить пункт навигации «Загружено» между «Скачать» и «Статистика»
- `templates/library.html` — новый. Группы по дням, каждая группа — список треков с `<audio controls>`, кнопкой скачать (link на `/file/{id}`), кнопкой удалить (POST на `/file/{id}/delete`)
- `templates/_library_recent.html` — мини-фрагмент для inline-блока на `/`
- `templates/index.html` — добавить блок `<div id="recent" hx-get="/library/fragment" hx-trigger="every 5s">` внизу формы
- `templates/stats.html` — убрать `<a href="/file/{id}">скачать</a>`, добавить секцию «топ устройств» (Chart.js bar по IP)

### Миграция БД
- Schema_version 1 → 2: ALTER TABLE добавляет 2 колонки. Старые строки получают `client_ip=NULL`, `deleted_at=NULL` — статистика прошлого не теряется, просто новой графики не будет для старых записей.

## Edge cases

- **Удаление файла, который ещё используется в `<audio>` другой вкладки**: браузер просто получит 410 на `/file/{id}` — это норм, мы не блокируем pessimistic-lock'ом.
- **Soft-delete через UI**: статистика по `success` использует фильтр `WHERE deleted_at IS NULL` для «текущих треков» и без него — для «исторических». Чтобы тренды не ломались.
- **Mini-panel «Загружено» на `/`**: показывает последние 5 `success AND deleted_at IS NULL`. Если меньше — показывает сколько есть. Если 0 — скрывается.
- **Per-IP с прокси (Caddy/Nginx)**: читать `X-Forwarded-For` если есть, иначе `request.client.host`. Брать первый IP из X-Forwarded-For (real client).
- **Длинные названия в плеере**: `truncate max-w-[40ch]` как и в queue (TD из v1).
- **/library с пустой БД**: empty-state «Пока ничего не скачано — перейди на главную».
- **Удаление трека, который сейчас качается**: нельзя, кнопки не показываем (только для `status='success' AND deleted_at IS NULL`).

## Success Criteria

- Можно зайти на сайт, скачать трек, нажать play на самой же странице (mini-panel) — звук пошёл без перехода на другую вкладку.
- Вкладка `/library` показывает все треки сгруппированные по дням; каждый день collapsible; в каждом — плеер + скачать + удалить.
- Удаление трека: файл исчез с диска (`ls /opt/yt2mp3/data/downloads/` подтвердит), `/stats` показывает уменьшенное число треков, исторические графики не сломались.
- `/stats` чистый — никаких кнопок «скачать», только цифры и графики. Новый чарт «топ устройств» рисуется по `client_ip`.
- Все 58 тестов зелёные. ruff clean.
- Deploy на 93.115.240.38: `git pull && docker compose up -d --build` подхватывает миграцию автоматически (schema_version 1 → 2 на startup).

## What I noticed about how you think

- Ты прямо в `taskv2` написал требования как чек-лист «1, 2, 3, …, ---, 5, ---, 6, ---, 7». Эти `---` — твой собственный способ выделить «отдельные подсистемы». Видно, что ты подсознательно уже сгруппировал: 1-4 это «работа со скачанным», 5 это «облако», 6 это «статистика», 7 это «UX оптимизация». Это правильная декомпозиция — на ней я и построил приоритеты (5 в отдельный релиз).
- Ты выбрал «А: аналитика по моим устройствам», когда я предложил A/B/C. Не пошёл по C (полный мультиюзер) — это taste-сигнал: не хочешь добавлять complexity, которая не решает твою задачу. Ровно по spirit'у v1.
- Требование 6 — «убрать из статистики возможность скачивать треки» — это тонкое замечание. Большинство людей оставило бы кнопки «потому что они там есть». Ты понял что `/stats` про статистику, а действия принадлежат другой вкладке. Это правильный design instinct — separation of concerns на уровне страниц.
- Требование 7 — «мини-Загружено внизу страницы Скачать» — кивок в сторону того, что ты замечаешь не только что у тебя нет фичи, но и что переключение вкладок раздражает. Это уже UX-зрелость, а не функциональный запрос.
